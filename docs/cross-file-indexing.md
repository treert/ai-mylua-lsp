# 跨文件索引与引用（讨论稿）

本文档集中讨论 **跨文件定义、类型、`require` 绑定、引用索引** 的数据结构与更新策略，便于与 [`architecture.md`](architecture.md) 总览拆分、**单独迭代定案**。需求级约定见 [`requirements.md`](requirements.md) §3.2.1（全局已见 + `local require` → `return`）。

**状态**：草案；冷启动索引策略、链式字段 hover、Lua table shape / Emmy 优先级、全局 table 特殊支持的总体方向已确定；持久化格式、references 语义等可在评审后继续收紧。

---

## 1. 冷启动索引策略（已定）

### 1.1 决议

冷启动采用 **方案 B：摘要驱动的混合式渐进索引**，作为当前稳定方向：

- **并行** 扫描工作区文件，生成每文件 **`DocumentSummary`**。
- `DocumentSummary` 产出后，**流式 merge** 到工作区聚合层（`GlobalShard` / `TypeShard` / `RequireByReturn` 等）。
- 查询层显式感知 **indexing state**；首轮全库完成前允许 **渐进可用但可能不完整**，完成后进入 **稳定工作区语义**。
- 热路径仍以 **摘要查询优先**，避免在 `goto` / `hover` 时做整库级即时分析。

这一定义兼顾了以下目标：

- **实现难度与可维护性**：不把系统锁死在“必须全库 barrier 后才可查询”的模型里，同时避免纯流式方案的过度状态复杂度。
- **冷启动性能**：接受工作区首轮扫描约 **3 秒** 的后台初始化窗口，但索引应尽快进入“部分可用”状态。
- **热路径性能**：`goto` / `hover` 等查询完成后，优先命中摘要与分片索引，避免随查询触发昂贵的全局计算。

### 1.2 备选方案对比

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **方案 A：严格两阶段** | 先完成全量 `DocumentSummary`，再统一 merge | 最易实现、最易维护、一致性最直观 | 冷启动期间跨文件能力几乎不可用；后续渐进索引与编辑期增量会更割裂 | **不选为主方案**；可作为实现早期的过渡形态 |
| **方案 B：摘要驱动的混合式渐进索引** | 并行摘要 + 流式 merge + 查询层区分索引状态 | 性能、可维护性、后续增量更新之间平衡最好；适合大工作区 | 比严格两阶段更复杂，需要定义 indexing state 与降级语义 | **选定** |
| **方案 C：纯流式即时索引** | 每个文件一完成就完全对外可见，查询始终基于当前部分索引 | 最早开始可用；大仓首屏体验最好 | 状态管理最复杂；结果波动更明显；测试与维护成本高 | **暂不选**；后续若有必要可向 B 演进 |

### 1.3 冷启动状态与查询语义

建议将查询层状态至少区分为：

- `Initializing`：文件集合已开始枚举，摘要与聚合层正在持续构建；查询可返回 **部分结果**。
- `Ready`：纳入索引范围的文件集合已完成首轮摘要生成与一次一致性 merge；查询结果视为 **完整工作区语义**。

在 `Initializing` 阶段：

- `goto definition`、`hover` 可返回部分结果，但实现需接受“尚未扫描到目标文件”导致的漏报。
- `workspace/symbol` 可提供渐进结果。
- `references` 语义应单独界定，避免默认给出“看似完整但实为不完整”的结果。

### 1.4 与编辑期增量的一致性

虽然用户编辑过程不属于“冷启动”本身，但本方案刻意与编辑期增量更新保持同一心智模型：

- 冷启动可视为“从空工作区状态开始，对全部文件做一次大批量差量导入”。
- 编辑期则是“基于已有索引，对少量受影响文件做小批量差量更新”。

这使得冷启动、保存更新、文件重命名、批量变更在架构上更统一。

## 2. 「数据库」类比与你的理解

**你的理解里正确的部分**

- **单文件信息相对独立**：每个 URI 可先解析成 **摘要（DocumentSummary）**，文件之间 **无解析顺序依赖**；在工程上可 **并行**（线程池、任务队列）生成摘要。
- **跨文件「视图」像数据库**：把多份摘要 **汇总** 成 **工作区级结构**（全局名 → 定义候选、`require` 反向边、类型表等），供 `goto` / `hover` / `workspace/symbol` / `references` 查询，**在概念上接近「内存里的逻辑数据库」**。

**需要细化的部分（不必「等全部文件解析完」才有跨文件能力）**

| 模式 | 说明 |
|------|------|
| **两阶段批处理** | 阶段 1：并行产出所有摘要；阶段 2：一次性 **merge** 成全局分片。**你的理解完全对应这种模式**；实现简单、易于推理一致性。 |
| **流式增量合并（推荐大仓）** | 每 **完成** 一个文件的摘要，立刻把其 **对全局层/`RequireByReturn` 的贡献** merge 进去。**不必**等全库扫完：索引 **逐步变完整**，早完成的文件已可跨文件解析（仍可能有「尚未入库文件」导致的漏报）。 |
| **编辑期** | 用户改一个文件时，只对 **该文件 diff → 差量更新** 聚合层，**不**重建整库。 |

结论：**单文件可并行；全局层 = 汇总层**。是否「必须全部解析完再建库**」是实现策略选择：批处理简单；流式 + 增量更贴合 **渐进索引** 与 [`requirements.md`](requirements.md) 里大工作区体验。

## 3. 语义模型概要（与架构一致）

与真实 Lua 加载不同，静态层约定如下（详见需求文档）：

- **全局已见**：工作区内各文件对 **全局** 的贡献进入 **合并视图**（遵守 `local`/块作用域）；**不要求**先 `require` 才能「看见」别号文件里的全局符号。
- **`local xxx = require("…")`**：路径可静态解析时，`xxx` **绑定**到目标文件的 **`return`**（及 Emmy/摘要近似）；维护 **反向索引** 供增量与 Hover。

## 4. 跨文件类型与名字解析（逻辑）

### 4.1 工作区「全局层」

- 对每个纳入范围的文件，抽取 **对全局环境有贡献** 的声明（`local` 不进全局；顶层赋值/函数声明等进入合并视图）。
- **同名冲突**：保留 **候选列表**（路径优先级、配置等 **显式策略**）；UI 展示歧义，避免静默选错。

### 4.2 `local xxx = require("…")` → 绑定到模块 `return`

- **条件**：形如 `local <name> = require(<可静态解析的字符串>)`（细节以文法为准）。
- **路径解析**：模块串 → 目标 URI（`?.lua`、别名等 **可配置**）。
- **语义**：`<name>` 对应目标文件 **`return`** 的模块值近似；**不**用此边推断「其它文件是否加载」。
- **反向索引**：`(目标 URI) → [(来源文件, 局部名), …]`。

### 4.3 类型名（Emmy）

- `---@class`、`---@alias` 等进入 **工作区类型表**；解析顺序建议 **本文件 → 工作区类型表（含冲突候选）**。
- `---@type Foo` 等仍可能多义。

### 4.4 解析标识符（定义 / Hover）简版流程

1. Lua 作用域（`local`、块、闭包）。
2. 若为 `local` 且属 §4.2 绑定 → 目标文件 `return` + 模块摘要。
3. 若为全局自由名 → 查全局合并表。
4. Emmy 类型名 → 本文件类型表 → 工作区类型表。

### 4.5 与动态现实的边界

- 非静态 `require`、拼接：不建 §4.2 绑定；全局仍靠 §4.1。
- `_ENV`、运行期改全局：配置或 unknown。
- 大工作区：摘要 + 分片 + 可取消后台任务。

## 5. 索引结构（「内存数据库」草图）

### 5.1 每文件摘要 `DocumentSummary`

对每个 URI，与 **版本或内容哈希** 绑定；除名字索引外，还需承载 **表达式类型传播** 所需事实：

- **全局贡献**列表（名称、类属、定义区间…）。
- **局部层**：`local`/形参/块；**含** §4.2 **绑定** `(局部名 → 目标 URI + return 指纹/区间)`。
- **类型表**：类型定义 + 类型名 **出现**区间。
- **函数摘要 `FunctionSummary`**：参数、返回类型、返回值形状、关联 Emmy 注解、必要的惰性重算标记。
- **局部类型事实 `LocalTypeFacts`**：关键局部变量当前可推出的类型候选及其来源（赋值、调用返回、字段访问等）。
- **表形状 / 字段事实 `TableShape` / `MemberFacts`**：字段名到字段类型、注解、定义位置的映射，用于成员访问链逐段解析。
- **可选**：值标识符出现列表（供 `references`）；可按名分桶压缩。

**AST**：LRU；跳转优先走摘要。

### 5.2 工作区聚合层

- **`GlobalShard`**：`全局名 → [候选定义…]`；更新时 **按名差分**（删旧文件贡献、插新）。
- **`TypeShard`**：`类型裸名 → [候选定义…]`。
- **`RequireByReturn`**：反向表；目标 `return`/Emmy 变更时失效依赖方缓存。
- **`FunctionReturnIndex`（可并入类型/符号分片）**：函数符号或可解析调用目标到返回摘要的映射，供表达式链查询快速命中。
- **可选 `Postings` / `symbolId → refs`**：`references` 速度与一致性权衡见 §5.

分片可按 **名字哈希/前缀** 切段，利于锁与并行。

### 5.3 单文件变更增量（`didChange` / 保存）

1. 重解析 → `S'`；与旧 `S` **diff**。
2. `GlobalShard` / `TypeShard`：对涉及名字 **删旧桩 + 插新桩**。
3. `RequireByReturn`：绑定增删；失效相关缓存。
4. 函数返回摘要、局部类型事实、字段形状：按受影响符号/表达式链差量失效。
5. 引用索引：按 postings 或 symbol 脏集更新。
6. 冲突排序：仅对 **受影响名字** 重算。

批量文件变更：合并、去抖、可取消。

### 5.4 `goto` / `hover` 热路径

- 局部：当前摘要 + 必要 AST。
- `local`+require：绑定表 **O(1)**。
- 全局名 / 类型名：**一次分片查找**。
- **链式字段 hover**：按“表达式 → 基础类型 → 字段类型 → 下一段字段类型”逐段解析；优先命中 `FunctionSummary`、`LocalTypeFacts`、`TableShape` / Emmy 字段信息，而非仅依赖名字表。

### 5.5 `textDocument/references`（已定）

#### 5.5.1 总体语义

- `references` 默认查找的是 **与当前光标同一决议的语义目标**，而不是所有同名文本出现。
- 变量名与字段名都需要先完成语义决议，再基于其身份查找读写引用。
- 这意味着：
  - **局部变量** 先区分作用域与闭包捕获关系；
  - **全局变量** 先区分普通全局、链式全局路径与 `GlobalNodeId`；
  - **字段** 先区分其宿主属于 Emmy 字段、普通 Lua table 字段，还是全局 table 字段。
- 标准 LSP `textDocument/references` 返回的是 **位置列表**，不直接携带 read/write 元数据；但服务端内部索引应保存这些信息，供过滤、排序和未来扩展使用。

#### 5.5.2 内部引用分类

- 第一阶段内部引用索引显式记录：
  - `read`
  - `write`
  - `readwrite` / `unknown`
- 内部索引统一收集 **声明/定义/读/写** 位点；最终响应 `textDocument/references` 时，严格按 LSP 的 `includeDeclaration` 参数裁剪结果。

#### 5.5.3 身份模型

`references` 的目标身份按语义类别区分：

- **局部变量**：按 `LocalSymbolId` 建模；闭包捕获沿用同一个 `LocalSymbolId`。
- **全局变量 / 全局链路**：同时保留两层身份：
  - 主身份：`GlobalNodeId`
  - 辅助索引：全局名 / 完整路径字符串
- **Emmy 字段**：同时保留两层身份：
  - 主查询身份：`TypeId + FieldName + DeclSite`
  - 展示/简化身份：`TypeId + FieldName`
- **普通 Lua table shape 字段**：同时保留两层身份：
  - 主查询身份：`TableShapeId + FieldKey + OriginSite`
  - 简化身份：`TableShapeId + FieldKey`
- **全局 table 字段**：同时保留两层身份：
  - 主查询身份：`GlobalNodeId + FieldKey + OriginSite`
  - 简化身份：`GlobalNodeId + FieldKey`

#### 5.5.4 `OriginSite` / 来源锚点

- 字段身份内部保留 **完整来源集合**，包括：
  - 注解来源
  - 首次写入来源
  - 其他补充来源
- 同时为外部查询选择一个 **主锚点**，用于稳定 identity、缓存键与增量失效。
- 主锚点优先级：
  - 显式注解 / 声明来源优先
  - 否则取首次把该字段引入该身份的写入位点

#### 5.5.5 候选策略

- 当光标位置存在多个候选身份时，`references` 默认只查 **最佳候选**。
- 但底层索引仍保留候选集合，不在查询前丢失信息。
- `references` 与 `goto definition` 共享同一套候选评分来源：
  - **显式类型来源优先**（Emmy 定义 / 显式注解 / 显式绑定链）
  - 纯 shape 推断结果优先级较低
- 与 `goto definition` 不同的是，`references` 在分数接近时更保守，更容易回退到 **多候选** 策略，避免把不同语义目标误并到一起。

#### 5.5.6 实现建议

- 朴素实现：`Postings[name]` + 语义过滤。
- 混合实现：`symbolId → refs` 懒惰维护；定义变更置脏 **引用簇**；大任务分段 + `$/cancelRequest`。
- 热路径上，建议优先解析当前光标的目标身份，再去 postings / symbol 索引中查找对应引用，而不是先做纯文本同名检索。

### 5.6 冷启动与持久化（可选）

- 第一阶段将**磁盘持久化缓存**做成配置项控制：
  - 默认启用 **`DocumentSummary` 级别缓存**
  - 用户可关闭为 **纯内存索引**
- 第一阶段**不默认缓存聚合分片**；聚合层可在启动时重建或渐进 merge。
- 推荐缓存模式：
  - `summary`：缓存每文件 `DocumentSummary`
  - `memory`：不使用磁盘缓存

#### 5.6.1 缓存失效规则（已定）

`DocumentSummary` 缓存失效至少绑定以下维度：

- **文件内容哈希**
- **grammar / LSP schema 版本**
- **关键配置指纹**

其中关键配置指纹仅纳入**会影响语义结果**的配置，例如：

- `require` 解析规则
- 工作区包含/排除模式
- 影响语义建模的功能开关

纯 UI 展示配置不必进入该指纹。

#### 5.6.2 按失效原因分级处理（已定）

- **文件内容变化**
  - 直接使对应文件的 `DocumentSummary` 失效
  - 重建该文件摘要，并按差量更新聚合层
- **grammar / LSP schema 版本变化**
  - 视为强失效
  - 放弃旧缓存批次，重新建立缓存
- **关键配置指纹变化**
  - 若配置影响语义结果，则按强失效处理
  - 第一阶段可直接清空受影响缓存并重建

#### 5.6.3 磁盘格式与缓存目录（已定）

- 第一阶段磁盘缓存格式采用**混合方案**：
  - **每文件一个 `DocumentSummary` 缓存单元**
  - **独立元信息 / 索引文件**，记录 schema 版本、配置指纹、缓存批次信息、URI 映射等
- 缓存目录支持两种归属：
  - **用户缓存目录**（默认）
  - **工作区内缓存目录**（可选）
- 默认放在工作区外的用户缓存目录，避免污染仓库并降低误提交风险。

### 5.7 多根工作区与 `exclude` 一致性（已定）

#### 5.7.1 多根工作区

- 多根工作区的索引边界由配置控制。
- **默认行为**：多个 root **合并为统一工作区索引**，以满足 workspace-wide 的定义、引用、符号能力。
- **可选行为**：允许按 root 隔离索引，适用于需要强隔离或同名冲突较多的场景。

#### 5.7.2 `exclude` 文件

- `exclude` 命中的文件默认**不进入工作区索引**。
- 但如果用户当前打开了该文件，仍允许对其做**临时单文件分析**，提供单文件级解析、局部跳转、基础 Hover 等能力。
- 打开中的 `exclude` 文件可以**读取**工作区内已建立的索引结果，用于跨文件解析与导航。
- 但该文件自身的分析结果默认**不回写**到工作区索引，避免反向污染全局语义。

## 6. 链式字段 hover（已纳入硬性需求）

### 6.1 需求约束

以下能力视为核心语义能力，而非可选增强：

```lua
local obj = some_outer_func()
local px = obj.pos.x
```

当光标停在 `x` 上时，Hover 需要能够沿下列链路求得字段信息：

1. `obj` 的类型来自 `some_outer_func()` 的返回类型或返回值形状；
2. `obj.pos` 的类型来自 `obj` 的字段事实；
3. `obj.pos.x` 的最终字段类型与注解来自下一层字段事实；
4. 若链路跨文件，应通过模块 `return` 摘要、函数返回摘要、工作区类型表继续传播。

### 6.2 选型对比

| 方案 | 核心做法 | 优点 | 缺点 | 结论 |
|------|----------|------|------|------|
| **A：注解优先，推断为辅** | 主要依赖 Emmy 注解，浅层局部推断 | 最易做、结果稳定 | 未注解代码很容易退化为 `unknown`，不满足链式 hover 的目标强度 | **不选** |
| **B：摘要驱动的强静态推断** | 在 `DocumentSummary` 中保存函数摘要、局部类型事实、字段形状；Hover 按表达式链逐段解析 | 能兼顾能力强度、查询速度、维护成本；适合与跨文件索引整合 | 需要额外设计类型传播与失效规则 | **选定** |
| **C：激进全程序推断** | 接近编译器式全程序约束求解 | 理论能力最强 | 复杂度、风险、误判成本都过高 | **暂不选** |

### 6.3 选定方向

链式字段 hover 采用 **方案 B：摘要驱动的强静态推断**，作为当前稳定方向：

- `DocumentSummary` 不仅记录“定义了什么”，还记录“值/函数/表达式可推出什么类型事实”。
- `hover` / `goto` 遇到成员访问链时，不只查名字表，而要执行 **表达式类型传播**。
- 类型来源优先级可概括为：**局部事实 → 函数返回摘要 / 模块 `return` 摘要 → Emmy 类型表 / 字段注解 → 结构推断 → unknown**。
- 第一阶段目标是支持“明显可静态分析”的强推断；对于动态 `__index`、复杂运行时分支、不可静态解析调用等情形，可显式降级为 `unknown` 或候选集合。

这意味着：`DocumentSummary`、聚合层、热路径查询都必须把 **函数返回值**、**局部变量类型传播**、**字段形状** 视为一等数据，而非附属信息。

### 6.4 第一阶段类型传播边界（已定）

- **分支合并**：`if/else` 默认对可达分支做 **候选并集**；在合并前先做 **轻量常量传播** 与 **死分支裁剪**。明显不可达的分支不进入结果。
- **循环**：`while` / `for` / `repeat` 第一阶段采用 **保守策略**；不追求复杂循环收敛分析，必要时返回保守候选或 `unknown`。
- **函数返回优先级**：`---@return` 等 **显式注解优先** 于函数体推断；函数体推断主要作为无注解时的补充。
- **元表**：第一阶段 **不支持** `metatable.__index` 参与字段链解析。
- **联合类型字段访问**：默认对每个候选分别解析字段，再合并结果；不因某个候选解析失败就直接把整体降为 `unknown`。
- **多返回值函数**：`FunctionSummary` 同时保留 **候选列表** 与 **合并后的总摘要**；Hover 默认先展示总结果，并附简短提示说明“存在多个返回候选”。
- **Hover 展示重心**：链式字段 Hover 默认以 **最终字段结果** 为主，不默认展开详细传播链。

### 6.5 Lua table shape（已定）

#### 6.5.1 核心建模

- **每个 Lua table 字面值都是一等语义对象**，应生成稳定的 `TableLiteralId` / `TableShapeId`。
- **主身份按字面值 AST 节点建模**；局部变量、字段、返回值都只是 **引用** 该节点。
- `FunctionSummary` 对返回的 table 值，应同时保留：
  - **导出 shape 摘要**，供查询热路径与跨文件消费。
  - **回指到源 table 字面值节点的引用**，供单文件内更强的追踪与解释。
- 普通 Lua table 的 shape 在 **单文件内可读写**；出了文件后只暴露 **只读摘要**，不传播可写语义。

#### 6.5.2 单文件内 shape 规则

- **空表 + 后续补字段** 属于正常 shape 构建路径，例如 `local t = {}; t.pos = {}; t.pos.x = 1`。
- 若两个局部变量引用同一个 table 节点，则后续写入都作用在 **同一个 shape** 上。
- 对 **直接可判定的重绑定**，需要精确更新别名关系，例如 `a = {}` 后，`a` 不再指向原节点。
- table shape 的目标是“**在方便实现的基础上提供最大静态收益**”，而非完整对象分析；更复杂的稳定接口应推荐用户改写为 EmmyLua 类型。

#### 6.5.3 字段写入与 shape 更新

- 以下写入可并入精确字段集：
  - `t.x = v`
  - `t["x"] = v`
  - `t[expr] = v`，但仅限 `expr` 能 **无歧义** 静态折叠成固定字符串。
- 数组风格字段也纳入 shape：
  - 若索引是可静态确定的 **固定整数**，则记录到 `indexedFields`。
  - 若索引无法静态确定为固定整数，则不记录精确槽位，只更新 `arrayElementType`。
- 同一字段多次赋值时，默认合并成 **联合类型**；若可静态判定旧赋值不可达，则不必并集。
- `t.x = nil` **不删除字段**，也不改变 shape 结构；只把字段标记为 **可为 nil**。
- “可能为 nil”的信息第一阶段先 **只在内部保留**，不急于在 Hover/诊断中广泛对外展示；仅在极高确定性的情形下，后续才考虑升级为显式诊断。

#### 6.5.4 递归 shape 与截断

- 返回表字面量支持 **递归记录嵌套字段 shape**，以支撑 `obj.pos.x` 这类链式字段 Hover。
- 第一阶段只限制 **最大递归深度**。
- 达到深度上限后，不直接退化为纯 `unknown`，而是保留 **`truncated shape`** 标记。

### 6.6 Emmy 与 table shape 的优先级（已定）

- **统一规则**：只要某个节点已绑定 **明确 Emmy 类型**，就从该节点开始 **完全切换到 Emmy 语义**；`table shape` 不再参与解析，也不作为补充展示。
- 这条规则同时适用于：
  - 全局 table 链路
  - 普通局部变量
  - 函数返回值
  - 字段访问链的中间节点
- 对全局 table 及其链式语句，只认 **显式注解** 触发 Emmy 绑定；不接受仅凭隐式推断切换全局节点语义。
- 认可的显式绑定来源包括：
  - `---@type`
  - 可明确把节点声明为 class/实例的注解关系
  - `---@return` 经由 **显式赋值链** 落到目标节点

#### 6.6.1 `---@class` 字段来源约束

`---@class` 的 field 维持强约束，仅认以下来源：

- 注释声明的字段
- 成员函数
- **可明确归属** 的 `self.name = expr`

其中 `self.name = expr` 仅在接收者可明确归属时生效：

- 若当前函数可明确识别为某个 `---@class` 的成员方法，则记到该 class。
- 若不是 class 方法，但 `self` 可明确解析到某个 Lua table 节点，则记到该 table 的 shape。
- 若无法明确知道 `self` 属于谁，则不生成字段语义。

#### 6.6.2 Emmy 语义下的诊断

- 一旦节点进入 Emmy 语义，字段读写都不再回退到 `table shape`。
- Emmy 类型语义下：
  - **字段存在但赋值不兼容**：默认诊断为 **`error`**，但允许用户配置降级。
  - **字段不存在**：读取与写入都默认报 **`unknown field`**，默认诊断为 **`error`**，但允许用户配置降级。

### 6.6.3 诊断总体策略（已定）

- 第一阶段诊断采用**平衡型**策略：
  - **Emmy 路径**默认严格，按强约束语义报错
  - **Lua 路径**默认保守，只对高确定性问题给出诊断
- 诊断语义与导航/类型传播语义保持一致：
  - 一旦命中 **明确 Emmy 类型**，就完全按 Emmy 路径处理
  - 未命中 Emmy 的对象，才按 **Lua `table shape` 路径**处理
- 配置层允许用户调整默认 severity，但不改变这两条路径的基础语义边界。

#### 6.6.4 Lua 路径诊断（已定）

Lua 路径第一阶段只对**高确定性**问题出诊断；一般性的动态性、候选不完整、普通 `nil` 风险默认只保留在内部语义层。

默认规则如下：

- **显式 `nil` 成员访问或赋值**：默认 **`error`**
  - 例如 `local x = nil; x.a = 1`
- **显式非对象值的成员访问或赋值**：默认 **`error`**
  - 例如 `local x = 1; print(x.a)`
- **开放 / 动态结构上的未知字段访问**：默认 **`warning`**
- **封闭 shape 上明确不存在的字段访问**：默认 **`error`**
- **字段赋值类型与当前静态 shape 明显冲突**：默认 **`warning`**
  - 例如 `local t = { x = 1 }; t.x = {}`
- **字段先被置为 `nil`，后续继续深层访问其子字段**：默认 **`warning`**
  - 例如 `local t = { x = { y = 1 } }; t.x = nil; print(t.x.y)`
- **联合候选里仍有合法路径支持当前字段**：默认 **不报诊断**

这条规则与前文保持一致：

- Lua 路径允许对象 shape 演化与联合类型存在
- 只有在静态上足够确定“这里很可能就是错的”时，才默认对外给出诊断

#### 6.6.5 Lua 路径诊断配置（已定）

- Lua 路径诊断第一阶段采用**默认简单分组，后续可扩展单规则配置**的方式。
- 建议先提供两组配置：
  - **Lua 高确定性错误**的 severity 配置
  - **Lua 保守提示**的 severity 配置
- 内部仍保留更细的诊断原因，后续若需要，可再扩展到按规则单独配置（如 `nilAccess`、`unknownField`、`shapeConflict`）。

### 6.7 全局 table 特殊支持（已定）

- 全局 table 作为 **跨文件共同定义** 的结构，需要特殊支持；多个文件里的链式写入都应注册到同一个工作区级 `GlobalTable`。
- 跨文件合并时，同时保留两层身份：
  - **逐段节点树**：如 `_G`、`_G.A`、`_G.A.B`
  - **完整路径索引**：如 `_G.A.B.C`
- 多个文件对同一全局路径补字段时：
  - 先做工作区级 **结构合并**
  - 保留 **来源文件/来源语句**
  - 若同一字段冲突，则保留 **多候选来源**
- 冲突结果在内部同时保留两层信息：
  - **合并后的联合类型 / 候选集合**
  - **候选来源列表**
- 全局 table 的“创建语句”采用较宽松识别：只要能静态确认某条语句是在 **创建或延续同一全局节点**，就可并入该 `GlobalTable`。
- 但一旦全局链路上的某个节点已显式绑定 Emmy 类型，就从该节点开始停止 `GlobalTable` 扩张，改走 Emmy 字段解析与赋值检查。

#### 6.7.1 全局同名冲突默认策略（已定）

- 当同一个全局名存在多个候选定义时，默认按**固定优先级选一个最佳候选**，但内部仍保留**完整候选列表**。
- 默认优先级采用**综合打分**，核心因素包括：
  - **显式定义强度**：Emmy 明确绑定、显式全局创建/赋值链优先于弱推断
  - **来源稳定性**：声明更完整、候选更少、语义更确定者优先
  - **路径 / 工程规则**：配置指定的优先目录、主工作区路径等
- 当两个全局候选分数接近时：
  - `goto definition` 尽量选择一个最佳候选
  - `hover` / `references` 更倾向回退到候选列表或多候选语义
- 这延续了本文整体原则：**内部保留完整候选信息；默认交互给出一个可用答案；在高歧义时优先保守。**

### 6.8 建议配置项（建议）

以下设置项与当前讨论结论直接相关，建议后续在扩展/LSP 配置中提供：

| 设置项 | 建议值 | 说明 |
|--------|--------|------|
| `lua.gotoDefinition.strategy` | `auto \| single \| list` | `auto`：有明显最佳候选则单跳，否则多目标；`single`：尽量直接跳最佳候选；`list`：优先展示候选列表。 |
| `lua.references.strategy` | `best \| merge \| select` | `best`：只查最佳候选；`merge`：多个候选身份的引用做并集；`select`：先选候选，再查对应引用。 |
| `lua.globalConflict.strategy` | `auto \| single \| list` | 全局同名冲突时的默认决议策略；`auto`：有明显最佳候选则单决议，否则回退候选列表。 |
| `lua.index.cacheMode` | `summary \| memory` | `summary`：缓存每文件 `DocumentSummary`；`memory`：只做内存索引。默认 `summary`。 |
| `lua.index.cacheLocation` | `user \| workspace` | 缓存目录位置；默认 `user`。 |
| `lua.workspace.indexMode` | `merged \| isolated` | 多根工作区默认 `merged`，允许配置为按 root 隔离。 |
| `lua.diagnostics.emmyTypeMismatchSeverity` | 默认 `error` | Emmy 字段存在但赋值类型不兼容时的诊断级别；允许用户降级。 |
| `lua.diagnostics.emmyUnknownFieldSeverity` | 默认 `error` | Emmy 类型下读取/写入不存在字段时的诊断级别；允许用户降级。 |
| `lua.diagnostics.luaErrorSeverity` | 默认 `error` | Lua 路径下高确定性错误的诊断级别，例如显式 `nil` 访问、显式非对象值成员访问、封闭 shape 未知字段。 |
| `lua.diagnostics.luaWarningSeverity` | 默认 `warning` | Lua 路径下保守提示的诊断级别，例如开放结构未知字段、shape 冲突、字段置 `nil` 后继续深层访问。 |

说明：

- `goto definition` 的“最佳候选”默认按 **显式类型来源优先** 排序：Emmy 定义 / 显式注解优先于纯 shape 推断结果。
- `references` 默认语义是“**同一决议的语义目标**”，不是所有同名文本出现；若需要更保守或更全面的行为，可通过 `lua.references.strategy` 调整。
- 全局同名冲突、缓存模式、多根工作区边界也均通过简单配置项暴露，优先保证默认行为清晰，再允许用户按项目习惯调整。
